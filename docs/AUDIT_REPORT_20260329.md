## 1. セキュリティ — 重大欠陥 (5.5/10)

### INFO（問題なし）: proxy.ts はNext.js 16で自動有効

Next.js 16では`middleware.ts`は非推奨となり、`proxy.ts`が後継として自動認識される。
`src/proxy.ts`にCSRF保護・セッションリフレッシュ・未認証リダイレクトが**堅実に実装済み**であり、
**追加のmiddleware.ts作成は不要**（作成するとビルドエラーになる）。

**proxy.tsの実装内容（既に自動的に有効）:**
- CSRF保護（Origin/Hostヘッダー検証、sec-fetch-site確認）
- Supabaseセッション自動リフレッシュ（JWT有効期限5分前に更新）
- 未認証ユーザーのリダイレクト（/admin→/login、/insurer→/insurer/login）
- 未公開機能のブロック（HIDDEN_ADMIN_PREFIXES）
- 静的アセットのスキップ

| ポータル | 保護方式 | リスク |
|---------|---------|-------|
| `/admin` | proxy（サーバーサイド）+ AdminRouteGuard（クライアント） | 低（二重保護） |
| `/agent` | proxy（サーバーサイド）+ AgentRouteGuard（クライアント） | 低（二重保護） |
| `/insurer` | proxy（サーバーサイド）のみ | **中（クライアント側ガードなし）** |
| `/customer` | Cookie検証（遅延） | 低 |

### ~~CRITICAL~~ → LOW（訂正済み）: Resend Webhook シークレット

`/api/webhooks/resend/route.ts` — コード上は`RESEND_WEBHOOK_SECRET`未設定時にスキップする条件分岐が残るが、
**本番環境ではシークレットが設定済み**であり実害なし。防御的コーディングとして未設定時に500を返す修正が望ましい。

### HIGH: Cron署名検証の実装バグ

`/lib/cronAuth.ts` — Vercel cron署名の検証で**空文字列をハッシュ**している。署名検証として機能していない。

### HIGH: テナントアクセス制御の不備

一部APIルートで`tenant_id`をパラメータとして受け取るが、呼び出し元がそのテナントの所有者であるか検証していない。LINEのWebhookルートが典型例。

### その他のセキュリティ懸念

- CSRF保護はproxy.tsに実装済み（Origin/Host検証方式）。middleware接続後に有効化される
- Admin APIルートにレートリミットが未適用
- 顧客ポータルの認証が「メール＋電話下4桁」のみ（脆弱）
- API鍵ローテーション戦略が未策定

### 評価できる点

- Zod によるバリデーション基盤は堅実
- Supabase RLS によるテナント分離は適切
- 公開エンドポイントにはレートリミットあり
- CSP/HSTS/X-Frame-Optionsなどのセキュリティヘッダーは適切
- Stripe Webhook署名検証は正しく実装
- ファイルアップロードのマジックバイト検証あり

## 2. DB/APIレイヤー — 不整合多数 (6.0/10)

### トランザクション非対応

Supabase PostgREST SDKは**トランザクションをサポートしない**。代理店承認（4〜5ステップ）やPOS決済など、複数テーブルを更新する処理が**非原子的**。中間ステップで失敗すると孤立データが発生する。

### APIレスポンス形式が不統一

247のAPIルートで**4種類以上のレスポンス形式**が混在：

```
Format A: { "agents": [...] }
Format B: { "ok": true, "item": {...} }
Format C: { "public_id": "...", ... }
Format D: { "error": "unauthorized" }
```

クライアント側の実装が脆弱になる原因。

### ページネーション未実装のエンドポイント

- `/api/admin/agents` — 全件返却（ページネーションなし）
- `/api/admin/invoices` — ページネーション不明
- `/api/agent/rankings` — ページネーションなし
- 20以上のGETエンドポイントでページネーション欠落

### バリデーション欠落

- 23以上のPOST/PUTルートでJSONパースエラーのtry-catchなし
- 注文作成、ドキュメント作成、請求書エンドポイントにZodスキーマ未適用
- 金額フィールドの型チェック不足

### その他のDB問題

- `job_orders.from_tenant_id = to_tenant_id`を防ぐ制約なし（自己参照注文が可能）
- 証明書テーブルの非正規化データ（車両情報）に同期トリガーなし
- ソフトデリート戦略なし（ほとんどのテーブルに`deleted_at`カラムがない）
- マイグレーションのロールバック手段なし（97マイグレーション全てが追記のみ）
- `pending_remote.sql`（6038バイト）が未コミット状態で残存
