# Ledra アーキテクチャリスク変更プラン

## 概要

本ドキュメントは、Ledra WEBアプリの監査で特定されたアーキテクチャ上のリスクに対する
段階的な改善計画を定義する。

---

## 1. モノリス肥大化への対策

### 現状
- 98,000行超のTypeScript/TSXコードが1つのNext.jsアプリに集約
- Admin/Agent/Insurer/Customer/Marketing/APIが同一ビルド
- ビルド時間増大、1バグが全体に影響するリスク

### Phase 1 (4月): ルートグループによる論理分離
- Next.js App Routerのルートグループ `(admin)`, `(agent)`, `(insurer)` を活用
- 各ポータルの `layout.tsx` でポータル固有の依存のみロード
- `next/dynamic` による遅延ロードを全ページに適用
- **見積もり: 3日**

### Phase 2 (5-6月): APIレイヤーの分離
- `/api/admin/*`, `/api/agent/*`, `/api/insurer/*` を独立したミドルウェアチェーンに
- 共通ユーティリティを `@ledra/core` パッケージとして抽出（Turborepo移行）
- Vercelのskew protectionを有効化
- **見積もり: 2週間**

### Phase 3 (Q3): マイクロフロントエンド化（必要に応じて）
- Insurerポータルを独立したNext.jsアプリとして分離
- 共有UIコンポーネントを `@ledra/ui` パッケージに
- Vercelのmonorepoデプロイ活用
- **見積もり: 1ヶ月**

## 2. データベースマイグレーション管理

- 101件のマイグレーションファイル

### 対策

   - 101ファイル → 1ベースライン + 新規分

2. **ロールバックスクリプト** (4月)
   - 各マイグレーションに対応するdownファイルを作成
   - CI/CDにマイグレーション検証ステップ追加

3. **ブランチ戦略** (5月)
   - Supabase Branchingの導入（Preview環境ごとにDB分岐）
   - PRごとのマイグレーション自動テスト

## 3. 外部サービス依存度の低減

### 現状のSPOF（単一障害点）

| サービス | 障害時の影響 | 対策優先度 |
|---------|-------------|-----------|
| Supabase (DB/Auth) | **全機能停止** | 最高 |
| Stripe | 課金不可 | 高 |
| Vercel | サイト停止 | 中 |
| Resend | 通知停止 | 低 |
| Upstash | レート制限無効 | 低（対策済） |

### 対策ロードマップ

#### Supabase障害対策 (5月)
- Read Replicaの導入（参照クエリの分散）
- Point-in-Time Recovery (PITR) の有効化
- 定期バックアップの自動化（日次）
- ヘルスチェックエンドポイント + アラート（既存cronを拡張）

#### Stripe障害対策 (5月)
- Webhook受信のキュー化（QStash経由で再試行保証）
- 決済障害時のグレースフル・デグラデーション
- Stripe Statusページの監視 + PagerDuty連携

#### メール障害対策 (6月)
- Resendフォールバックとして AWS SES を設定
- メール送信の非同期キュー化（QStash）

## 4. サーバーサイドキャッシュ戦略

### 現状
- キャッシュなし（全リクエストがDBに直接アクセス）
- Upstash Redisはレート制限のみに使用

### 導入計画

#### Phase 1: ホットデータキャッシュ (4月)
```
Upstash Redis を活用して以下をキャッシュ:
- ダッシュボードサマリー: TTL 60秒
- テナント設定: TTL 300秒
- メニューアイテム一覧: TTL 300秒
- プラン情報: TTL 3600秒
```

#### Phase 2: ISR活用 (5月)
- マーケティングページ: `revalidate = 3600`（既に設定済み）
- 公開証明書ページ `/c/[public_id]`: `revalidate = 60`
- 料金ページ: `revalidate = 86400`

#### Phase 3: Edge Caching (6月)
- Vercel Edge Configを活用した機能フラグ管理
- CDN Cache-Controlヘッダーの最適化（静的APIレスポンス）

## 5. セッション管理の最適化

### 現状
- 全リクエストでSupabase Auth セッションリフレッシュ（HTTP call）
- レイテンシ増加 + Supabase API負荷

### 対策 (4月)
1. セッショントークンのローカル検証（JWT署名検証のみ）
2. リフレッシュは有効期限切れ間近（残り5分以下）の場合のみ
3. 実装: `proxy.ts` のセッションチェックロジック修正

## 9. パフォーマンス最適化（スケーラビリティ監査より）

### 9.1 ダッシュボード集計のSQL化（緊急）

**現状**: `/api/admin/dashboard-summary` がJS側で全証明書を取得しカウント。
50k件超でパフォーマンス劣化。

**対策**:
```sql
SELECT
  COUNT(*) FILTER (WHERE status = 'active') as active_count,
  COUNT(*) FILTER (WHERE status = 'void') as void_count,
  COUNT(*) FILTER (WHERE status = 'draft') as draft_count
FROM certificates
WHERE tenant_id = ? AND created_at >= ?;
```

### 9.2 レシート番号採番のシーケンス化

**現状**: `pos_checkout()` RPCが `COUNT(*)` でレシート番号を生成。
同時決済で競合リスクあり。

**対策**: PostgreSQLシーケンスに置き換え。

### 9.3 画像最適化パイプライン

**現状**: 証明書画像がSupabaseストレージから無加工で配信。
サムネイル生成・フォーマット変換なし。

**対策**:
- アップロード時にリサイズ・WebP変換（Vercel Edge Function or cron）
- サムネイル(400px)・中サイズ(1200px)・オリジナルの3段階保存
- CDN Cache-Controlヘッダー最適化

### 9.4 バンドルサイズ分析

**対策**:
- `@next/bundle-analyzer` 導入
- `@react-pdf/renderer`, `stripe`, `googleapis` の遅延ロード徹底
- CIにバンドルサイズ閾値チェック追加
