## 0. PoC の目的と「成功した状態」

目的は **「技術が動くことを見せる」ではなく「トヨタ担当者が社内で話を通せる材料を作る」こと**。

**成功した状態** は以下 3 つが揃った状態と定義する：

1. **画面の成功**: トヨタ担当者が「この車は Ledra でコーティング施工済み、Polygon で証明されている」という `/v/{vin}` ページを上司に見せられる。「URLを見せる」「Polygonscan を開く」の 2 アクションで完結する。
2. **数字の成功**: 「PoC 期間中に N 台の車両パスポートが発行された / M 件の証明書がアンカーされた / 閲覧 K 回」という持ち帰り資料が出せる。
3. **論理の成功**: 「外部のスタートアップが既にここまで動いていて、自前で作るより組んだ方が早い」という事実で社内稟議が通る。

**失敗の定義（先に決めておく）**:
- PoC 開始 30 日経って **発行証明書数 ≥ 5 件、Polygon Tx 成功率 ≥ 90%** に届かないなら、運用フローに問題あり → 軌道修正会議を即招集。
- PoC 終了時に **「数字は出たが社内決裁が動かない」状態**になったら、担当者単独依存（社内チャンピオン不足）の典型。次回からは複数階層接触に切替（`toyota-negotiation-strategy.md` §社内チャンピオン）。

## 2. PoC の 3 フェーズ

### フェーズ 0 — 社内準備（〜2 週間、トヨタ接触前）

> 「明日トヨタの会議に呼ばれても今日中にデモできる状態」を作るためのフェーズ。

#### Step 0-1: Polygon アンカリングを Mainnet で稼働

詳細は `docs/polygon-anchoring-deployment.md`。要点だけ抜粋：

1. **テストネットで先行確認**（既に終わっていれば飛ばす）
   - Amoy で `LedraAnchor` をデプロイ → faucet で test POL を取得 → 数枚アンカー → Polygonscan で `isAnchored` を確認。
2. **Mainnet コントラクトをデプロイ**
   - 同じ `LedraAnchor.sol`、専用署名鍵（PoC 中はローテーション不可なので慎重に発行）
   - デプロイトランザクションを Polygonscan で控えておく（後で証跡として使う）
3. **Vercel 本番 env 設定**
   ```
   POLYGON_ANCHOR_ENABLED=true
   POLYGON_NETWORK=polygon
   POLYGON_RPC_URL_POLYGON=<Alchemy/Infura/QuickNode の本番 RPC>
   POLYGON_CONTRACT_ADDRESS_POLYGON=0x...
   POLYGON_PRIVATE_KEY=0x...   ← Vercel の Secret として保存
   ```
   per-network env が効くので、`POLYGON_NETWORK=polygon` を残したまま `_AMOY` を併設しておくと過去 Amoy データの verify も生かせる（`src/lib/anchoring/providers/polygon.ts:7-13`）。
4. **POL 残高を 0.5 POL 以上に積む**
   - PoC 中の見込みアンカー量 ≈ 200 件 × 平均 4 画像 × 0.001 POL = 0.8 POL を上限見積もり
   - 残高アラート（Vercel cron もしくは Polygonscan ウォッチ）を仕込む
5. **本番で 1 件アンカー → Polygonscan で `Anchored` イベント確認**
6. **既存証明書のバックフィル**（PR-6 相当）
   ```bash
   # qstash polygon-backfill を一度キック
   curl -X POST https://<prod>/api/qstash/polygon-backfill \
     -H "Upstash-Signature: ..." -d '{}'
   ```
   バッチ完了後に `select count(*) from vehicle_passports` で件数を確認。PoC 開始時の「ベースライン数字」になる。

**Mainnet cutover 当日のチェックリスト**:

- [ ] Amoy で動作確認済み
- [ ] Mainnet コントラクト Etherscan/Polygonscan で「verified」状態
- [ ] 署名鍵を 1Password / Vercel Secrets に保管、鍵ローテ手順をドキュメント化
- [ ] RPC URL は有料プラン（無料 RPC は本番でレート制限に当たる）
- [ ] `POLYGON_ANCHOR_ENABLED=true` をデプロイ前後で確認（ロールバック可能な PR で）
- [ ] 残高 ≥ 0.5 POL、残高低下時 Slack 通知
- [ ] 失敗時のフォールバック方針を決めておく（"Amoy に一時切り戻し" or "アンカーをスキップして証明書だけ発行"）

#### Step 0-4: 弾薬（数字・第三者バリデーション）の補強

`toyota-negotiation-strategy.md` §交渉前に準備すべき「弾薬」と重複するが、PoC 着手時点で **会議で出せる数字** を作る。

| 数字 | 取得方法 | 目標値 |
|------|----------|--------|
| Polygon アンカー済み証明書数 | `select count(*) from certificate_images where polygon_tx_hash is not null` | 1,000 件以上 |
| 車両パスポート発行 VIN 数 | `select count(*) from vehicle_passports` | 100 VIN 以上 |
| 参加施工店数 | `select count(*) from tenants where ... active` | 10 店以上 |
| 累計証明書発行数 | `select count(*) from certificates` | 5,000 件以上 |
| 月次増加トレンド | 直近 3 ヶ月の発行件数推移 | 右肩上がり |

これらが揃わない場合は **PoC 開始を 1〜2 ヶ月遅らせて数字を作る** のも選択肢。空のプラットフォームをトヨタに見せても説得力が出ない。

#### Step 0-5: 内部チームのドライラン

トヨタ会議の 1 週間前までに、Ledra 社内で **完全デモ通し稽古** を実施：

- 営業役（CEO or 事業開発）が話す
- 社内 3〜5 名が「トヨタ担当者役」で質問を投げる（パートナーシップ提案書の「FAQ」に沿った質問が中心）
- 録画して見直す。詰まる箇所、専門用語の説明、デモが落ちた時のフォールバック対応を確認
- 「3 分動画 + 5 分質疑」「3 分動画 + 25 分質疑（30 分会議想定）」「フル 60 分」の 3 パターンで練習

#### Step 1-2: 初回コンタクトの実施手順

**① 事前リサーチ（接触 3 日前）**
- LinkedIn で接触先部署の中堅 PM・エンジニアを 5〜10 人特定
- 過去 3 ヶ月の発信内容を確認（ブロックチェーン関連の話題に反応している人を優先）
- 共通の知人がいないか調査（MOBI / J-Startup / IPA 未踏 経由のリファラルが理想）

**② 初回メッセージ（接触当日）**
- LinkedIn DM が第一選択（メールより返信率が高い）
- `toyota-outreach-email.md` の該当テンプレを **300 字以内に圧縮** して送る
- **提案書 PDF や長文は添付しない**。3 分動画の URL のみ
- 件名は「{動画 URL} を見ていただけませんか」レベルにシンプル化

**③ 返信があったら（〜3 日以内）**
- 「15 分の Web 会議」を即提案。日時候補は 3 つ、自分側は完全フレキシブル
- 会議ツールは先方都合（Zoom / Teams / Google Meet どれでも）
- 会議 URL とアジェンダ（10 分デモ + 5 分質疑）を返信に同封

**④ 返信がない場合（送付から 5 営業日経過）**
- 1 回だけリマインド（「もし関心がなければスルーで OK です」と書く）
- それでも反応なければクローズ。同じ会社の別部署 / 別人物に切替

#### Step 1-3: 初回ミーティング（15〜30 分）の設計

**話す順番（厳守）**:

1. **冒頭 30 秒**: 「コーティング・PPF の施工証明をブロックチェーンで発行するプラットフォームです」「現在 N 店 / M 件の証明書が稼働しています」
2. **デモ動画 3 分**: 事前に渡してあっても会議で再生する。共通体験になる
3. **デモ動画後 1 分**: 「Woven City（or KINTO・TBL）の文脈ではこういう使い方が考えられます」と提案を 30 秒で
4. **質疑応答 5〜10 分**: 相手の質問を全て聞く。答えに詰まったら「持ち帰って 2 営業日以内に回答します」で OK
5. **クロージング 1 分**: 「次回はライブデモも含めて 1 時間ほど時間をいただけますか」「あるいは PoC の具体的な話をしたいので、御社の決裁プロセスを教えていただけますか」

**話さないこと（重要）**:
- 技術スタックの詳細（C2PA、ABI、Merkle、ハッシュ関数の名前など）
- 契約形態・料金体系
- 将来構想（「いずれは保険、いずれは融資、いずれは中古車…」）
- Ledra の資本構成・調達状況

→ 全部「次回」に回す。最初の 30 分は **「やってみる気になるか」だけが論点**。

**よくある質問への返答カード**:

| 質問 | 返答 |
|------|------|
| 「ブロックチェーンって本当に必要？」 | 「中央集権 DB でも記録は残せますが、第三者（保険会社・査定担当者）が **Ledra 経由なしで** 真正性を確認できる手段はブロックチェーンが現状ベストです」 |
| 「Polygon である理由は？」 | 「ガス代が安く、トランザクション速度が実用レベル。Ethereum Mainnet だと 1 アンカー数千円かかります。トヨタ MON は Avalanche ですが、施工証明ハッシュはチェーン非依存なので将来のクロスチェーン記録も可能」 |
| 「個人情報は大丈夫？」 | 「ブロックチェーンには SHA-256 ハッシュのみ。写真本体・氏名・電話・価格はオフチェーン。PII マスク方針は `docs/dpa-template.md` に明文化済み」 |
| 「他の自動車メーカーとも話してる？」 | 「日産・ホンダ系の窓口にも並行アプローチしています。先行優位を提供できるのは最初の 1 社だけ」 ← **必ず言う**（`toyota-negotiation-strategy.md` §レバー① 競合圧力） |
| 「規模が小さくない？」 | 「アンカリング基盤は Vercel / Supabase / Polygon の組み合わせなので、規模では大手 SaaS と同等のスケーラビリティ。`docs/AUDIT_REPORT_20260503.md` でセキュリティ監査済み」 |

### フェーズ 3 — 評価と本番交渉（PoC 終了後 1 ヶ月）

> 目標: PoC の数字を武器に本番採用・拡大の話をする。継続見送りでも次の打ち手を確保する。

#### 3-1. 評価指標（PoC 終了時点）

| 指標 | 目標値 | 計算方法 | 意味 |
|------|--------|----------|------|
| 発行証明書数 | 50 件以上 | `count(certificates where tenant_id = woven-city-poc)` | 「実際に使われた」証拠 |
| Polygon アンカー成功率 | 95% 以上 | `count(certificate_images where polygon_tx_hash is not null) / count(certificate_images)` | システムの信頼性 |
| 車両パスポート発行 VIN 数 | 30 台以上 | `count(distinct vin_code_normalized from vehicle_passports created during PoC)` | 「Ledra 証明付きの車」が存在する |
| パスポートページ閲覧数 | 100 回以上 | アクセスログ（PostHog / Vercel Analytics） | 実際に参照された回数 |
| 施設スタッフの継続利用率 | PoC 3 ヶ月間継続 | 各スタッフの月次最終ログイン日 | 離脱なく使い続けた事実 |
| Polygonscan で `Anchored` イベント確認可能 | 100% | スポットチェック 10 件 | 第三者検証可能性 |
| NFC タグ発行数 | 20 枚以上 | `count(nfc_tags where written_at during PoC)` | 物理タグ × デジタル証明の繋ぎ込み実証 |

**数字に達しない場合の解釈**:
- 発行 < 20 件 → 運用課題（スタッフが Ledra を覚えなかった、もしくは施設の施工頻度自体が少なかった）
- アンカー成功率 < 90% → 技術課題（RPC ノードの安定性、鍵管理、ガス価格高騰）
- 閲覧 < 30 回 → マーケ課題（QR コードを渡すだけでは顧客が見ない、誘導動線を改善）

#### 3-2. PoC 結果プレゼン（終了後 2 週間以内）

トヨタ担当者と決裁ライン（部長・本部長）に向けたプレゼン。
**30 分構成**、PDF 30 ページ以内。

```
[1-3] エグゼクティブサマリ（数字、結論、3 つの選択肢）
[4-8] PoC で実証されたこと
   - 「Woven City 内 N 台の車両に対してリアル運用ができた」
   - 「Polygon 上に X 件の改ざん不可能な証明書が存在する」
   - 「スタッフ Y 名が 3 ヶ月離脱なく使い続けた」
[9-14] 発生した問題と解決した結果
   - ガス代スパイク時の対応、鍵管理、UI 改善
   - トヨタ側からのフィードバック取り込み事例
[15-20] 本番採用時の運用イメージ
   - Woven City 全施設展開のスケジュール案
   - ディーラー展開の絵姿
   - TBL エコシステムへの統合可能性
[21-25] 本番交渉の選択肢 A / B / C（次節）
[26-28] リスクと対策
[29-30] 次のステップ・連絡先
```

#### 3-3. 本番交渉での 3 つの提案

**オプション A: Woven City 全施設への展開**

- 対象: Woven City 内の全モビリティ関連施設（整備・洗車・コーティング・PPF）
- 期間: 2026 Q4 から 12 ヶ月契約、自動更新
- 費用: 月額 30 万円（5 施設想定、施設追加で従量制）
- 条件: Ledra の「Woven City 公認施工証明プラットフォーム」マーケティング活用を可とする代わりに価格優遇
- 期待効果: Woven City のブランドで Ledra の信用が一気に立ち上がる

**オプション B: TBL エコシステム統合**

- 対象: Toyota Blockchain Lab の車両履歴データベースとの API 統合
- 期間: Phase 1 (6 ヶ月) 共同設計 → Phase 2 (12 ヶ月) 実装
- 費用: 共同開発、Ledra 側は API レイヤー構築コストを負担
- 条件: MOBI DID 互換性を確保、TBL 認定マークの付与を受ける
- 期待効果: トヨタの車両ライフサイクルデータの「アフターマーケット領域」を Ledra が埋める正式パートナー化

**オプション C: ディーラー展開の橋渡し**

- 対象: トヨタ系列ディーラー（地域 1 〜 10 拠点での先行導入）
- 期間: 個別契約（ディーラー単位の SaaS サブスクリプション）
- 費用: 既存の Ledra プラン（月額 5〜15 万円 / 店舗）
- 条件: トヨタ本社のリファラル協力（紹介状・パイロット成功事例の公開）
- 期待効果: トヨタ系列での実績が他自動車メーカー系列ディーラーへの展開ドライバになる

**3 つを同時に提示** し、相手が一番ハードルが低い選択肢に飛びつきやすい状態を作る。

#### 3-4. 「継続見送り」シナリオの処理

PoC が終わったあと、トヨタ側が「本番採用は見送る」と判断する可能性は 40〜60% ある。
その場合に Ledra として確保すべきもの：

1. **見送り理由のヒアリング** — 「予算」「決裁プロセス」「他に優先課題があった」「機能不足」のどれか必ず聞く。これが次のプロダクト改善・営業戦略の最重要インプット
2. **公開可能性の交渉** — 「PoC を実施した事実」だけでも公開できるか確認。"Toyota Woven City で実施した PoC では XX 件の証明書が発行された" と言えるだけで Ledra のセールス価値が跳ねる
3. **継続的な関係維持** — 担当者個人との関係は残す。半年後に状況が変わって再開する可能性がある
4. **競合への切り替え** — 日産・ホンダ系列への切り替えを並行で進めていれば、見送りでも Ledra 側のダメージは小さい（`toyota-negotiation-strategy.md` §レバー① 競合圧力）

**継続見送り後の Ledra 内部処理**:
- 失敗ではなく「市場検証」として総括（Wins/Losses 振り返り）
- PoC 中に得た技術的学び（Polygon ガス代運用、NFC 実地検証、cross-tenant データ集約）は全社の財産として記録
- メディア向けには「Toyota Woven City での実証実験を完了」と公表（先方の許可を得てから）

## 3. デモシナリオ集（営業会議で使う台本）

PoC 開始前から終了後まで、相手の関心に合わせて使い分ける 3 シナリオ。

### シナリオ A: 「この車のコーティングは本物です」（基本シナリオ）

**対象**: Woven by Toyota 担当者、KINTO 事業開発

**流れ**:
```
[施工スタッフ]
1. /admin/certificates/new を開く
2. 車両 VIN を入力（例: JT2BF22K1W0066983 / トヨタ車実在フォーマット）
3. 施工内容「セラミックコーティング」と写真 3 枚を登録
4. 「証明書を発行」をクリック
   → QR コード付き PDF が生成される
   → バックグラウンドで Polygon Tx が発行される（30 秒以内）

[車両オーナー]
5. QR コードをスマホでスキャン
   → /c/{public_id} が開く（施工証明書）
   → 「この車両の全履歴を見る →」バッジをタップ
   → /v/{vin} が開く（車両パスポート）

[第三者: 保険会社・査定担当者]
6. /v/{vin} を直接共有 or 検索 で確認できること:
   - 「コーティング施工: 2026-04-26 | Woven City モビリティセンター」
   - 「Polygon Tx: 0x... (Polygonscan で検証可能)」
   - 他にも施工記録があれば時系列で一覧表示
```

**強調するポイント**: 「第三者が誰でも Ledra のサイト経由なしで真正性を確認できる」「これがトヨタが今持っていないもの」。

### シナリオ B: 「中古車査定が変わる」（中古車・下取り文脈）

**対象**: KINTO、トヨタ販売店、トヨタモビリティ

**流れ**:
```
[ある車両のオーナーが下取りを依頼]
1. ディーラー査定担当者が車両の VIN をスマホで入力
2. /v/{vin} を開く
   → 過去 5 年間に施工した記録 7 件が時系列で表示
     - 2022: 新車納車後の標準コーティング
     - 2023: PPF 施工（ボンネット・フロントバンパー）
     - 2024: 飛び石跡の鈑金塗装（左フロントフェンダー）
     - 2025: 2 年目の再コーティング
     - 2026: 軽微なドア凹み修理
   → すべて Polygon Tx で改ざん検知済み
3. 査定担当者が判断:
   → 「施工履歴がしっかり残っている → 推定残価率 +5%」
   → 「軽微な事故歴があるが、修理証明があるのでマイナス幅を抑える」
```

**強調するポイント**: 「車両の履歴が査定額に直接反映される」「Ledra がない車両は『不明な過去』として割引されるリスク」。中古車ビジネスに直結する金額の話なので、決裁者が動きやすい。

### シナリオ C: 「保険金詐欺を防ぐ」（損保連携文脈）

**対象**: KINTO（保険連動）、保険会社（既に Ledra ポータルあり）

**流れ**:
```
[保険会社の事故対応担当]
1. 事故車両の修理見積もりが上がってきた
2. 修理工場が「過去にコーティングしていたので塗装の重ね塗りが必要」と主張
3. 担当者が /v/{vin} で確認:
   → 「該当時期にコーティングの施工記録なし」
   → 修理工場の主張に矛盾あり → 追加調査
   OR
   → 「2024 年 5 月にコーティング施工済み、Polygon Tx で検証可能」
   → 主張に整合性あり → 通常処理
```

**強調するポイント**: 「保険会社のコスト削減」「車両情報の改ざんを防ぐ」「Ledra のあるエコシステムは社会的に有用」。

## 4. リスクと対策

| # | リスク | 兆候 | 対策 | 対応責任 |
|---|--------|------|------|----------|
| R1 | Polygon Mainnet のガス代高騰で Tx 詰まり | 1 件のアンカーに 30 秒以上 / RPC エラー連発 | Amoy testnet に一時切戻し（per-network env が効くので runtime cutover 可能）。PoC 開始時の合意書に「テストネット併用」を明記しておく | エンジニア |
| R2 | トヨタ側スタッフが Ledra を使ってくれない | 週次の発行件数がゼロ | Day 0 オンボーディング時に「1 件発行できるまで」横にいる。A4 マニュアル冷蔵庫貼付。週次レポートで「他の施設では平均 N 件 / 週」を出して比較圧力 | 営業 |
| R3 | トヨタ法務がデータ管理に懸念 | NDA / PoC 合意書のドラフトに長時間の修正要請 | `docs/dpa-template.md` を先に提示。「ブロックチェーンにはハッシュのみで PII はオフチェーン」資料を初回会議で配布 | 営業＋法務 |
| R4 | PoC 後に「継続見送り」 | 中間チェックでフィードバックが空虚 / 担当者の上司に届かない | 中間チェック時点で兆候があれば、決裁ライン上位への直接アプローチを検討。並行で日産・ホンダ系列の話を進めておく | CEO |
| R5 | 担当者が異動 | 9 月以降の人事異動シーズン | 最初から 2〜3 名と関係を作る。引き継ぎ書を Ledra 側から準備して渡す | 営業 |
| R6 | 個人情報の意図せざる漏洩 | Polygon にウォレットアドレス以外の PII が乗る、`/v/{vin}` で本来マスクすべき情報が見える | `getPassportData` の PII マスクテスト（`src/lib/passport/__tests__/getPassportData.test.ts`）を CI で守る。`/v/{vin}` で表示する情報を初回会議でスクリーンショット付きで開示 | エンジニア |
| R7 | Polygon コントラクト署名鍵漏洩 | Vercel Secrets / 1Password でのアクセスログ異常 | 単一署名鍵 + 鍵ローテ手順を `docs/disaster-recovery.md` に追記。秘密鍵は会社代表者のハードウェアウォレットに切り替え検討 | エンジニア |
| R8 | RPC プロバイダ障害 | Alchemy / Infura の障害ステータス | 2 社の RPC を切り替え可能にする（`POLYGON_RPC_URL_POLYGON` を 2 系統運用）。1 系統障害時は手動切替手順を整備 | エンジニア |
| R9 | デモテナントの本番混入 | `woven-city-poc` テナントが顧客リスト・売上に紛れ込む | 本番ダッシュボード集計から PoC テナントを除外（`tenants.is_demo_or_poc` フラグ）。月次レポートで PoC 行のみ別建て | エンジニア＋経営管理 |
| R10 | 「他社にも同じことをやっている」と疑われる | トヨタ担当者が日産・ホンダ系列の進捗を直接質問 | 並行アプローチは正直に伝える（隠すと信用毀損）。ただし「具体的な相手先」は秘匿。フレーム B（オープン化）で対応 | CEO |
| R11 | Polygon ネットワーク自体の技術的非互換 | Polygon ハードフォーク / PoS → AggLayer 移行による旧 Tx の検証不能 | 検証関数は `verifyAnchor(sha256, network)` で per-network 切替可能。長期保証は契約書に「ブロックチェーン技術側の変更には合理的な追随努力」と明記 | エンジニア |
| R12 | Woven City 自体の運営方針変更 | Woven City 内のプロジェクト凍結 / 担当部署の解散 | 並行で KINTO・ディーラーの話を継続。Woven City 単独依存にしない | CEO |

## 5. Ledra 内部のリソース・コスト計画

PoC を回すためのリソースを社内で見える化する。3 ヶ月 PoC の想定。

### 5-1. 人的工数

| 役割 | 担当 | 工数 (人日 / 3 ヶ月) |
|------|------|----------------------|
| 営業・コミュニケーション | CEO or 事業開発 | 30 人日 |
| エンジニア（オンコール・改善対応） | エンジニア 1 名 | 20 人日 |
| カスタマーサクセス（オンボーディング・週次レポート） | サポート 1 名 | 25 人日 |
| データ分析（中間・最終レポート） | データ 1 名 | 5 人日 |
| 法務・契約書修正 | 外部弁護士 | 10 時間 |
| **合計** | | **約 80 人日 + 弁護士 10 時間** |

### 5-2. 直接コスト

| 項目 | 金額 (3 ヶ月) | 備考 |
|------|---------------|------|
| Polygon ガス代 | 〜5,000 円 | 1 アンカー約 0.5 円 × 1,000 件想定 |
| Polygon RPC（Alchemy 有料プラン） | 〜30,000 円 | 月 1 万円 × 3 ヶ月 |
| Vercel / Supabase の追加リソース | ほぼ無視可能 | 既存契約内で吸収 |
| デモ動画制作（外注なら） | 〜200,000 円 | 内製なら工数のみ |
| 法務（PoC 合意書修正） | 〜100,000 円 | 既存顧問契約に追加 |
| 出張費（Woven City 訪問 月 1〜2 回） | 〜120,000 円 | 静岡県裾野市、新幹線往復 |
| **合計** | **約 50 万円** | サブスクリプション売上を放棄する機会損失は別 |

### 5-3. 機会損失の確認

- PoC のために CEO / エンジニア / サポートの稼働が他案件に向かない期間がある
- その間、他の有償顧客向け機能リリースが遅れるリスク
- → PoC 中もスプリント枠の 20% は通常開発を守る運用ルールを設定

```
2026年 5月（残り）
  ├─ Polygon Mainnet 本番投入（POLYGON_ANCHOR_ENABLED=true）

  ├─ 1 社で PoC 合意（NDA + PoC 範囲書締結）
  ├─ Day 0 オンボーディング
  └─ PoC 開始

2026年 7〜9月
  ├─ PoC 実施（3 ヶ月）
  ├─ 週次レポート / Day 45 中間チェック
  └─ Day 75 から本番交渉の準備

2026年 10月
  ├─ PoC 終了、最終レポート作成
  └─ 本番交渉プレゼン

2026年 Q4
  └─ 本番契約合意 or 継続見送り
```

このスケジュールは **「全てがうまくいった場合」** であり、初回アウトリーチで全く反応がない場合は 6 月の段階で接触先リストを倍に広げる。
