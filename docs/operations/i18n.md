# i18n convention

**Status**: foundation only. Existing Japanese strings are intentionally
**not** migrated; this doc defines the pattern for **new** code so the door
to a second locale is open without a future big-bang refactor.

## Layout

```
messages/
  ja.json         ← source of truth (Japanese-first product)
  en.json         ← placeholder, mirrors ja keys
src/lib/i18n/
  index.ts        ← public API
  locales.ts      ← supported locale registry
  messages.ts     ← t() / getTranslator() — server-side translator
  negotiate.ts    ← Accept-Language → Locale
src/lib/api/responseI18n.ts  ← apiValidationErrorT / apiUnauthorizedT / …
```

## Adding a new translation

1. Add the key under `messages/ja.json` (source of truth).
2. Add the same key to `messages/en.json` with the English copy. Keys without
   an English value fall back to Japanese at runtime, but missing entries
   are surfaced in code review as a sign the locale is incomplete.
3. Use a dot-namespace per surface area:
   - `errors.*` — API error messages (rate_limited, invalid_payload, …)
   - `customer.*` — public customer-portal copy
   - `admin.*` — admin dashboard copy
   - `email.<template>.*` — outbound email subject/body fragments
4. Keep keys snake_case, dot-segmented, and stable. Renaming a key is a
   breaking change for any cached client copy or future translation tooling.

## Using translations on the server

```ts
import { negotiateLocale, t } from "@/lib/i18n";

export async function POST(req: Request) {
  const locale = negotiateLocale(req);
  if (!parsed.success) {
    return apiValidationError(t("errors.invalid_payload", locale));
  }
  // …
}
```

…or, for routes that produce many error branches:

```ts
import { getTranslator, negotiateLocale } from "@/lib/i18n";

const tr = getTranslator(negotiateLocale(req));
return apiValidationError(tr("errors.missing_field", { field: "email" }));
```

For routes that almost always return one of the standard error keys, the
`responseI18n` helpers do both steps in one call:

```ts
import { apiValidationErrorT, apiUnauthorizedT } from "@/lib/api/responseI18n";

if (!caller) return apiUnauthorizedT(req);
if (!parsed.success) return apiValidationErrorT(req, "errors.invalid_payload");
```

## What about UI strings (React)?

Out of scope for the foundation phase. When we want to localize a page:

1. Pick the smallest self-contained surface area (e.g. the public certificate
   page).
2. Move that surface's strings into `messages/{ja,en}.json`.
3. Resolve the locale at the route entry (server component) and pass it down,
   or wire `next-intl` for client components on that surface only.
4. Don't try to "lift" UI strings out of pages that aren't being localized
   yet — it adds churn and import noise without value.

## Variable interpolation

`{name}` placeholders, no other formatting. Same syntax as next-intl's plain
strings, which means callers can swap `t()` for `useTranslations()` later
without rewriting templates.

```json
{ "errors.missing_field": "{field}は必須です。" }
```

```ts
t("errors.missing_field", "ja", { field: "email" });
// → "emailは必須です。"
```

For complex pluralization or date/number formatting, do **not** invent ad-hoc
syntax — switch that specific call to next-intl's `useFormatter` /
`getTranslations` (already installed) and document the migration.

## Locale negotiation

`negotiateLocale(req)` parses `Accept-Language` and returns a supported
`Locale`. URL-segment routing (`/[locale]/...`) is **not** wired yet —
adding it would invalidate every existing URL, so it stays off until a
genuine bilingual deployment is on the table.

If you need an explicit override (e.g. user preference stored on their
profile), pass the `Locale` directly to `t()` instead of relying on the
header.

## Existing Japanese copy

Hardcoded Japanese strings throughout `src/` are intentionally untouched.
Migrating them in bulk would risk silent regressions in working flows (PDF
generation, emails, error funnels) without delivering English support yet.
The plan is to convert opportunistically — when a file is already being
edited for another reason, lift its strings into `messages/ja.json`.

## Tests

`negotiateLocale()`. `src/lib/api/__tests__/responseI18n.test.ts` covers the
API response wrappers. Both run as part of `npm test`.
