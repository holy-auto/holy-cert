## 0. なぜ損保ジャパンが Lighthouse か

経営チームの判断 (2026-05-13):

- **意思決定速度が速いと想定** — DX 推進部門が明確に存在し外部協業に前向きとされる (公開情報ベース)。⚠️ **当社からの接点は未確立。「直接接点あり」は選定根拠から除外。「意思決定速度が最速」は未検証の仮説**
- **指定工場ネットワーク (約 5,000 店)** が業界最大級 → 1 社目接続で店舗 ARR の指数拡大シナリオ
- **保険業務の SaaS 利用に前向き** — 競合 (Shopmonkey 等) より先に押さえれば 12〜18 ヶ月リード
- **DX 案件の単価が他 3 社より明示的** (¥500 万〜1,000 万/月 帯)
- **損保ジャパン日本興亜時代からの ADAS/事故車修理データ蓄積** がある → 当社の証明書データと突合する素地が業界で最も整っている

> ⚠️ **重要な前提整理 (2026-05-17 追記)**: 上記は「**損保ジャパンを Lighthouse 標的に選ぶ**」という *経営の意思決定* であって、**損保ジャパンとの関係性・接点を意味しない**。2026-05-17 時点で当社から損保ジャパンへの接点は **未確立**。
> したがって本ドキュメントの T-90 〜 T+90 タイムラインおよび 90 日撤退ラインは「**接点獲得後**」に始まる時計であり、**現在地は §3 フェーズ0 (NDA 締結前 / そもそも接点獲得前)**。
> 接点獲得の主チャネルは **Plug and Play Japan Insurtech 応募** (`docs/pnp-japan-insurtech-application.md`、SOMPO がアンカーパートナー) + 直接コールドの並行。PnP を単一障害点にしない。

他 3 社 (東京海上 / 三井住友海上 / あいおいニッセイ) は **PoC 合意フェーズのパイプライン**
として並行アプローチ (これも現時点で接点なし)。接点獲得後、Lighthouse 接続の 30 日後を目処に NDA 締結を狙う。

| # | 状況 | 判定 | 閾値根拠 |
|---|---|---|---|
| 1 | NDA 締結から **90 日** を超えても PoC 範囲書が確定しない | **縮小**: 2 社目並行を最優先に切替 | 日系大手 B2B SaaS の標準 PoC 着手リードタイムが 60-90 日。これを超えると意思決定が止まっている兆候 (`docs/direction-research-loot-drop.md` §2.7) |

| 3 | 月額 **¥300 万** を下回る価格を要求された | **撤退判断**: ARR 設計 (§3 ¥3 億分) が崩れる | §14.4 walk-away pricing で算出: 運用コスト ¥180 万 + 機会費用 ¥150 万 = ¥330 万。これ未満は事業赤字 |
| 4 | 損保ジャパン側担当が **3 ヶ月で 2 名以上** 交替 | **縮小**: 担当再立ち上げまで投資抑制 | 大企業の組織再編 / 異動シーズン頻度 (4 月・10 月) を考慮。2 回の引き継ぎを 3 ヶ月で行うと商談が実質ゼロリセット |
| 5 | 競合 (国内・外資) が同等の機能で **先に PoC 開始** した | **データ取り直し**: 撤退判断材料を 2 週間で集める | 並走するとリソース 1.5 倍、勝率 30% 程度に低下。差別化が成立しないなら他 3 損保にピボット |
| 6 | フェーズ 2 (接続前準備) が **3 ヶ月以上遅延** | **縮小**: 他 3 損保並行展開を優先 | 当社の機会費用月 ¥150 万 × 3 ヶ月 = ¥450 万。並行 3 社の Lighthouse 化のほうが期待値高い |
| 7 | 損保ジャパン経営層が **6 ヶ月以上** 「保留」 | **延期**: 3 ヶ月クールダウン | DX 予算は四半期サイクル。6 ヶ月の保留は予算未承認の高い shorthand |
| 8 | 加盟店 onboarding が T+180 で **30 店舗未満** | **縮小**: カスタマーサクセス再設計 | §7 マイルストーン (T+90 で 30 店) の 2 倍遅れ。事業前提が崩れている |

### 1.4 撤退時の経済合理性チェック

「撤退すべきか継続すべきか」の意思決定で迷ったら、以下の計算で機械的に判断:

```
継続価値 (CV)  = (期待 ARR × 成約確度) - (残り投入コスト)
撤退価値 (EV)  = 並行 3 社のうち 1 社を Lighthouse 格上げした場合の期待 ARR

if CV < EV : 撤退
if CV ≥ EV : 継続
```

**例: 損保ジャパン NDA から 6 ヶ月、PoC 未着手の場合**

```
継続価値 (CV)
  期待 ARR: ¥6,000 万 / 年
  この時点の成約確度: 25% (NDA 後 6 ヶ月で停滞は重い兆候)
  残り投入コスト: ¥1,500 万 (CEO + BizDev + エンジニア 半年分)
  CV = ¥6,000 万 × 25% - ¥1,500 万 = ¥0
  → 期待値ゼロ

撤退価値 (EV)
  並行 3 社のうち 1 社 (東京海上) を格上げ
  期待 ARR: ¥6,000 万 / 年
  この時点の成約確度: 40% (NDA 段階に進めた他社平均値)
  残り投入コスト: ¥2,000 万 (再立ち上げ)
  EV = ¥6,000 万 × 40% - ¥2,000 万 = ¥400 万
  → 期待値プラス

判定: CV < EV → 撤退して東京海上に切替
```

### 2.0 各フェーズ期間の根拠

```
フェーズ 0 (T-90 → T-60, 30 日): NDA 締結前
  - 損保ジャパン側 社内稟議 (NDA 単体) は通常 2-4 週間 → 30 日が現実線
  - 初回 MTG 設定リードタイム 2-3 週間 + 2 回目 MTG 1-2 週間 = 4 週間
  - これより短くすると損保ジャパン担当が「急かされた」と感じる
  - これより長くすると当社の機会費用が増大 (¥150 万 / 月)

フェーズ 1 (T-60 → T-30, 30 日): PoC 合意
  - PoC 範囲書 法務協議は通常 3-4 週間 (NDA より複雑だが NDA で慣れた後)
  - SLA 数値協議は 1-2 週間 (経営判断含む)
  - 並行作業可能なので 30 日で完了可能
  - 法務協議が長引けば §4.5 エスカレで弁護士直接 MTG

フェーズ 2 (T-30 → T-7, 23 日): 接続前準備
  - ペネトレ実施 + 報告書 = 2 週間 (発注先により 1-3 週間)
  - DPA 法務協議 = 2-3 週間
  - SAML IdP 連携 = 1 週間 (情報受領後)
  - 業務フロー 5 ケース テスト = 1 週間
  - これらを並行で実行して 23 日が物理最短

T-7 → T 0 (1 週間): 最終確認 + 準備
  - 全項目最終チェック 2-3 日
  - リハーサル 1-2 日
  - バッファ 2 日 (不具合修正用)

T 0 → T+7 (1 週間): 本番運用立ち上げ
  - 毎日 §6.4 デイリーチェックで監視強化期間
  - 想定外不具合の発見と修正の余裕

T+7 → T+30 (23 日): 加盟店 30 店オンボーディング
  - 5 店舗 × 5 日 (1 日 1 店舗 onboarding) × 5 ターン = 25 日
  - 並行可能なので 23 日で 30 店舗 (CS 体制依存)

T+30 → T+90 (60 日): 安定運用
  - SLA 達成 / KPI 安定を確認する最小期間 (Salesforce 推奨 SOC2 観測期間も 90 日)
  - この期間で「使われ続けるか」のサインを掴む

T+90 (Go/No-Go G4): 2 社目並行展開判断
  - 1 社目 90 日運用 = 1 社目テンプレ化完了タイミング (§20.1)
  - ここで G4 通過しないと並行 3 社の機会費用が膨張
```

→ **総期間 T-90 〜 T+90 = 6 ヶ月**。これが §20.2.1 で書いた 1 社目 6 ヶ月の中身。
これ以上短縮するには「並行作業を増やす」しかなく、当社リソース上限を超える。

### 4.3 PoC 範囲書 (合意書) の構造

PoC を「本番運用前のリハ」と位置づける。`docs/toyota-poc-plan.md` の構造を保険会社向けに改造:

```
PoC 範囲書に含める内容 (最小限)
1. PoC 期間: 60 日 (フェーズ 2 = 接続前準備期間と同義)
2. 対象業務: [損害サービスの修理見積照会 1 業務 のみ]

5. データ: PoC データは PoC 終了 60 日後に物理削除
6. 秘密保持: NDA 第◯条の範囲で
7. 本番採用の義務はなし (これが法務通過の鍵)
8. 撤退条項: 両者いつでも 14 日前通告で撤退可
```

#### 4.3.1 各数値の根拠

| 項目 | 値 | 根拠 |
|---|---|---|
| PoC 期間 | **60 日** | 日系大手の標準 PoC 期間 (60-90 日)。60 日は意思決定が間延びしない最短。`docs/toyota-poc-plan.md` の Woven City PoC が 3 ヶ月、保険会社は判定が早いため 60 日に圧縮 |
| 対象業務 | **1 業務 限定** | スコープ拡大は失敗パターン。`docs/direction-research-loot-drop.md` §3 教訓 #5「広く薄く同時に追って完成しない」を回避 |
| 対象店舗 | **5 店舗** | 1 店舗だと業務量不足で評価不能、10 店舗超だと CS 工数が PoC で潰れる。5 店舗 × 月 20 件/店 = 100 件/月 で統計的に意味のある最小サンプル |
| 無償期間 | **60 日** | 月額 ¥500 万 × 2 ヶ月 = ¥1,000 万 の機会費用。本番接続成功確度を 50% で見て期待損失 ¥500 万 ≈ 1-time 投資 §14.4.1 の 1/3 = 損益許容範囲 |
| データ削除 | **PoC 終了 60 日後** | バックアップ保持 30 日 + 監査対応猶予 30 日 = 60 日。即時削除は監査要求があった際に対応不能 |
| 撤退条項 | **14 日前通告** | 大企業の意思決定サイクル (週次経営会議 2 回分) で経営合意 → 通告まで完了できる最短 |

**意図的に「本番採用義務なし」を明記**: これがないと損保ジャパン法務が動かない。
ただし PoC が成功した場合の「本番接続契約への移行」を §◯ で明示しておく
(「妥当な対価で交渉する努力義務」レベル)。

## 9. SSO / IdP 連携 — 損保ジャパン IdP 別詳細手順

### 9.1 IdP 種別の特定

損保ジャパン側で使われている IdP は **事前ヒアリング必須**。日系大手の典型は以下:

| IdP | 確率 | 特徴 |
|---|---|---|
| Microsoft Entra ID (旧 Azure AD) | 高 | M365 と同梱、企業 IT で標準 |
| Okta | 中 | 外資寄りの大企業 |
| OneLogin / Ping | 低 | 特殊用途 |
| 自社 SAML (内製) | 中 | 古くから情シスが内製 |
| 既存の Sompo SSO 基盤 | 中 | 損保ジャパン HD 全体の統制下 |

### 9.2 必要情報 (損保ジャパンから取得)

```
[必須]
- IdP のメタデータ XML URL (推奨) or ファイル
- IdP の Entity ID
- 利用ドメイン (例: @sompo-japan.co.jp、@sompo-japan.jp など複数候補)
- NameID format (推奨: emailAddress)
- 必須属性: email, name (Supabase が user_metadata に格納)
[任意]
- SCIM 自動プロビジョニングを使うか (使う場合は SCIM bearer token を当社が発行)
- IdP 側のグループ/属性で role を制御するか (現状未対応、ロードマップ)
- 多要素認証 (MFA) は IdP 側で強制するか (推奨: 強制)
- セッション有効時間
- IP allowlist (損保ジャパン社内 NW のみ等)
```

### 9.3 Microsoft Entra ID (Azure AD) の手順

```
損保ジャパン情シス側
  1. Entra ID → Enterprise applications → New application
  2. "Supabase" または "Custom SAML app" を作成
  3. 以下を設定:
     - Identifier (Entity ID): https://<project>.supabase.co/auth/v1/sso/saml/metadata
     - Reply URL (ACS):       https://<project>.supabase.co/auth/v1/sso/saml/acs
     - Sign-on URL:           https://app.ledra.co.jp/auth/sso/start?domain=sompo-japan.co.jp
  4. Attributes & Claims:
     - email (Source attribute: user.mail)
     - name  (Source attribute: user.displayname)
  5. メタデータ XML を Download → 当社共有

当社側
  6. Supabase Dashboard → Authentication → SSO → Add provider
  7. Type: SAML 2.0 / Metadata file をアップロード / Domains: sompo-japan.co.jp
  8. tenants.sso_required = true 設定 (DB)
```

### 9.4 Okta の手順

```
損保ジャパン情シス側
  1. Okta admin → Applications → Browse App Catalog → "SAML 2.0" カスタム
  2. SAML 設定:
     - Single sign on URL:   https://<project>.supabase.co/auth/v1/sso/saml/acs
     - Audience URI:         https://<project>.supabase.co/auth/v1/sso/saml/metadata
     - Name ID format:       EmailAddress
     - Application username: Email
  3. Attribute Statements:
     - email, name
  4. メタデータ URL を当社共有

当社側
  5. Supabase Dashboard → SSO → Add provider → Metadata URL 入力
  6. tenants.sso_required = true 設定
```

### 9.5 自社 SAML (内製) の手順

```
損保ジャパン情シス側で SAML 2.0 IdP を立てている場合
  1. IdP 側に Service Provider (SP) として Supabase を登録
     - Entity ID: https://<project>.supabase.co/auth/v1/sso/saml/metadata
     - ACS URL:   https://<project>.supabase.co/auth/v1/sso/saml/acs
     - NameID:    emailAddress
  2. 必須属性: email, name
  3. IdP メタデータ XML を Export → 当社共有

当社側
  4. Supabase に登録
  5. テストアカウントで疎通確認
注意点:
  - 内製 IdP は時間ずれ (clock drift) に弱い。NTP 同期確認必須
  - 内製 IdP は証明書の有効期限が短いことがあり、半年に 1 回ローテが必要
  - 障害時の窓口を IdP 担当者 として明示してもらう (§16)
```

### 9.6 SCIM 自動プロビジョニング

> 当社実装: ⚠️ Roadmap (Lighthouse 1 社目契約後に着手予定 ~ 3 週間)
> Lighthouse では「初回ユーザ手動招待 + IdP 側 deactivate でセッション切断」で運用開始可能

将来的に対応する場合:
- 損保ジャパン IdP (Entra ID / Okta) に SCIM endpoint を登録
- SCIM bearer token は Ledra 側で発行 (1 IdP 1 token)
- ユーザ作成・削除・属性更新が当社 `tenant_memberships` に自動反映
- グループ → role マッピング (例: `sompo-insurer-admin` → `admin`)

### 9.7 IdP 障害時のフォールバック

| 状況 | 対応 |
|---|---|
| IdP がダウン (~ 数時間) | 損保ジャパン側で復旧待ち / 当社は監視継続 |
| IdP がダウン (> 4 時間) | 一時的に `tenants.sso_required = false` に戻し password sign-in 許可 (損保ジャパン情シス合意の上) |
| IdP 証明書が期限切れ | 損保ジャパン情シスに即時連絡 / 当社側で旧証明書を一時許容 |
| メタデータ URL 変更 | 当社で Supabase 側を更新 |

### 9.8 検証チェックリスト (フェーズ 2)

- [ ] `@sompo-japan.co.jp` のテストアカウントでパスワード sign-in を試み → SSO 案内画面にリダイレクトされる
- [ ] SSO 経由で sign-in 後、`/insurer` にアクセスできる
- [ ] IdP 側でユーザを deactivate → 次のセッションリフレッシュで Ledra も deactivate
- [ ] Sentry に `insurer_id` タグが付くことを確認 (`setSentryInsurerContext`)
- [ ] MFA を IdP 側で強制している場合、初回 sign-in で MFA challenge が表示される
- [ ] 複数ドメイン (`@sompo-japan.co.jp` / `@sompo-japan.jp` 等) で sign-in が動く
- [ ] SLO (Single Logout) — 損保ジャパン側で sign-out 時に Ledra も sign-out (任意)

- 監査ログ: 全リクエストを `audit_logs` に記録
- ログ保持: 1 年 (個情委ガイドラインに準拠)

中央値 (本表)
  2,500 店 (50%) → 店舗 ARR 月 ¥8,750 万 = 年 ¥10.5 億 → ARR ¥6 億達成

上振れケース (Tier A 90% / B 65% / C 30%)
  3,150 店 (63%) → 店舗 ARR 月 ¥1.10 億 = 年 ¥13.2 億

= 想定の幅広い (¥7-13 億) ため、§3 ARR 設計の店舗 ¥6 億は中央値クリア前提

## 14. 課金モデル・アップセル設計

> 価格は「ARR 目標から逆算した数字」だけで提示すると商談で詰められた瞬間に崩れる。
> このセクションでは **(a) 価値ベース (bottom-up)** / **(b) ARR ベース (top-down)** /
> **(c) 競合比較 (comp)** / **(d) 当社コスト (walk-away)** の 4 方向から数字を導き、
> 損保ジャパンが「なぜこの価格?」と聞いてきた時に **どの方向の説明でも答えられる** 状態を作る。

### 14.0 価格決定の 4 方向アプローチ

```
        Top-down (ARR ¥3億 / 保険社 3-5社 → ¥500-1000万 / 月)
           ↑
Comp ── 価格レンジ ── Value-based
  (国内 InsurTech SaaS    (Sompo にとっての年間
   月 ¥200-500万        ROI ¥3-5 億 → 10-25%)
   外資 ¥1000万+)          → ¥250-1000万 / 月)
           ↓
        Walk-away (当社運用コスト + 機会費用 → 最低 ¥300 万 / 月)
```

**4 方向が重なる帯 = ¥500-1,000 万 / 月**。
ここが提示の中心レンジ。

### 14.1 (a) 価値ベース計算 — Sompo が Ledra から得る金銭価値

損保ジャパンの自動車保険事業の概数から、Ledra が削減/創出する価値を bottom-up で積む。
**ヒアリング (§3.4 Q1〜Q5) で実数を取って差し替える** 前提のドラフト。

#### 14.1.1 修理見積照会の作業効率化

```
損保ジャパン 自動車保険シェア          : 約 23-25% (4 大の一角)
業界全体 自動車保険事故対応件数        : 約 400 万件 / 年 (推計)
└─ うち損保ジャパン                   : 約 100 万件 / 年
   └─ うち修理関連 (Ledra スコープ)   : 約 70 万件 / 年 (70% 想定)
      └─ うち指定工場経由             : 約 30 万件 / 年 (30%、Ledra 加盟店割合と一致させる)

1 案件あたりアジャスター作業時間
  - 現状: 平均 60-90 分 (見積照会 + 確認 + 承認)
  - Ledra 導入後: 40-60 分 (証明書 + 車両履歴 を一発参照)
  - 短縮効果: 20 分 / 案件 (中央値)

人件費換算:
  - アジャスター時給 ¥3,500 (年収 ¥600 万 / 1,700h 換算)
  - 短縮価値: 30 万件 × (20/60)h × ¥3,500 = ¥3.5 億 / 年
```

これを **3 年で全 5,000 店フル展開** した場合、年間 ¥10 億超の人件費圧縮効果。
ただし PoC 期間 (5 店舗) では数十万円規模にしか効かない。
**「3 年で ¥10 億削減」の絵を提示しつつ、初年度の課金は 5,000 店フル展開時の 5-10%** とする ロジックが説明可能になる。

#### 14.1.2 不正請求検知

```
損保ジャパン 自動車保険 元受保険料 : 約 ¥4,500 億 / 年 (推計、HD 統計)
└─ うち修理関連 支払保険金        : 約 ¥2,000 億 / 年
   └─ 不正請求 推定割合            : 1-3% (業界推計、公的統計薄)
      └─ 不正請求金額              : ¥20-60 億 / 年

Ledra による検知率改善:
  - Polygon アンカリングで施工写真/見積の改ざん検知
  - 車両パスポートで「過去施工と矛盾する見積」の自動検出
  - 改善率想定: 5-10% (国内損保のクレーム審査自動化事例から)
  - 削減効果: ¥1-6 億 / 年
```

不正請求は数字を断言しにくいが、**保険会社内部の SIU (Special Investigation Unit) は明確な KPI** として持っているので、ヒアリングで「年間の不正請求検知額 / 検知率」を聞き出してから具体数字を載せる。

#### 14.1.3 解約抑制 / NPS 向上 (中長期)

```
損保ジャパン 自動車保険 契約件数 : 約 1,500-2,000 万件
└─ 平均年保険料                  : 約 ¥45,000 / 件
└─ 解約率 (現状)                : 8-10% / 年 (業界平均)

事故対応満足度向上 → 解約率 0.1% 削減:
  - 1,500 万件 × 0.1% × ¥45,000 = ¥6.75 億 / 年 (保険料維持効果)
  - ただし因果関係の証明は困難、PoC では計測不能
```

これは "副次効果" として提示。価格根拠の主軸にはしない。

#### 14.1.4 戦略価値 (定量化困難)

- 指定工場ネットワークのデジタル化リーダーシップ → 他 3 社との競争優位
- 中古車市場での車両履歴データ蓄積 → 損保ジャパン HD グループ内クロスセル (損保ジャパン日本興亜パートナーズ等)
- OEM (トヨタ・日産) との将来データ連携の入り口

→ ¥億単位の戦略価値だが、契約交渉では「副次価値」として位置づけ、価格主軸は §14.1.1〜14.1.2 の合計。

#### 14.1.5 価値合計とプライシングルール

| 効果 | 初年度 (5 店舗 PoC + 100 店舗オンボーディング) | 3 年目 (5,000 店フル) |
|---|---|---|
| 作業効率化 | ¥3,500 万 〜 ¥7,000 万 | ¥7-10 億 |
| 不正請求検知 | ¥3,000 万 〜 ¥6,000 万 | ¥1-3 億 |
| 解約抑制 | (計測不能) | ¥1-3 億 (推定) |
| **合計** | **¥6,500 万 〜 ¥1.3 億** | **¥9-16 億** |

B2B SaaS の **value-based pricing 標準ルール = 創出価値の 10-25%**:

| シナリオ | 価値の % | 月額換算 |
|---|---|---|
| 保守的 (10%) | 初年度 ¥650 万 〜 ¥1,300 万 / 年 | **¥54-108 万 / 月** |
| 標準 (15-20%) | 初年度 ¥1,000 万 〜 ¥2,600 万 / 年 | **¥83-217 万 / 月** |
| 強気 (25%) | 初年度 ¥1,600 万 〜 ¥3,250 万 / 年 | **¥135-271 万 / 月** |

**初年度単独で見ると ¥500 万 / 月 は明らかに高い**。が、契約は 3 年単位で結ぶ前提なので、

| 3 年累計価値 (¥9-16 億 → 中央 ¥12 億) × 価格 % | 3 年契約額 | 月額換算 |
|---|---|---|
| 10% | ¥1.2 億 | ¥333 万 / 月 |
| 15% | ¥1.8 億 | ¥500 万 / 月 |
| 20% | ¥2.4 億 | ¥667 万 / 月 |
| 25% | ¥3.0 億 | ¥833 万 / 月 |

→ **3 年契約 + 段階値上げ前提で ¥500-833 万 / 月 が value-based の中央レンジ**。
これが価格根拠の主軸。

### 14.2 (b) ARR 逆算 — 当社が必要とする金額

`docs/ledra-goals-strategy-2026-05.md` §2.2 の 3 階建て ARR 設計から:

```

├─ 店舗サブスク     : ¥6 億   (1,400 店 × ¥35,000/月)
├─ 保険会社課金     : ¥3 億   (3-5 社合計)  ← Lighthouse はここ

**保険会社 ¥3 億 / 年 を 3-5 社で達成** する場合の 1 社あたり配分:

| シナリオ | 社数 | 1 社平均 / 年 | 1 社平均 / 月 |
|---|---|---|---|
| 3 社で達成 | 3 | ¥1 億 | ¥833 万 |
| 4 社で達成 | 4 | ¥7,500 万 | ¥625 万 |
| 5 社で達成 | 5 | ¥6,000 万 | ¥500 万 |

ただし **Lighthouse (1 社目) は基準を作る = 他社より高くてよい**。
理由: 業界初接続、最大の指定工場ネット (5,000 店)、リファレンス価値。

→ **Lighthouse は ¥833 万 / 月 以上** が ARR 設計上の目標値。
2 社目以降は ¥500-700 万 / 月 で揃える。

技術 RPO (実質) vs 契約 RPO (バッファ):

### 15.2 SLA 違反時のペナルティ — クレジット率の根拠

```
99.0% 〜 99.5% : 当該月の課金額の 5% を翌月クレジット
98.5% 〜 99.0% : 当該月の課金額の 10% を翌月クレジット
98.0% 〜 98.5% : 当該月の課金額の 20% を翌月クレジット
< 98.0%        : 当該月の課金額の 50% を翌月クレジット + 経営説明会
```

**業界相場との比較**:

| サービス | 99% 違反 | 95% 違反 | 上限 |
|---|---|---|---|
| AWS Compute SLA | 10% | 25% | 25% |
| Azure SLA | 10% | 25% | 100% |
| Salesforce | 個別交渉 | 個別交渉 | 上限明示なし |
| **当社 (Lighthouse)** | **5%** | **20%** | **50%** |

当社は AWS / Azure より穏やか (5-50%) に設定。理由:
- 損保ジャパンが受け取るクレジット価値より、当社の運営継続性のほうが両社の利益
- 50% Cap がないと、長時間障害 1 回で当社が事業継続困難になる (上限保護必須)

**経済影響のシミュレーション** (Lighthouse ¥500 万 / 月の場合):

```
月次達成 99.5% → クレジット ¥0
月 1 回の小障害 (達成 99.3%) → クレジット ¥25 万 (5%)
月 1 回の大障害 (達成 98.8%) → クレジット ¥50 万 (10%)
月 1 回の重大障害 (達成 98.2%) → クレジット ¥100 万 (20%)
月 1 回の致命的障害 (達成 97.5%) → クレジット ¥250 万 (50%, 上限)
```

### 15.3 Error Budget の活用

`docs/slo.md` §「Error Budget 消費ルール」に従い、損保ジャパン経由でも統一運用:

| 消費率 | アクション | 根拠 |
|---|---|---|
| < 50% | 通常運用 | 残バッファが十分、新機能リリース可 |
| 50-80% | 新機能リリース時に SLO 影響を計測 | 当該月内の追加違反が SLA を割る確率 30% 超 |
| 80-100% | リリース凍結 / 修正のみ | 追加違反すれば確実に SLA 違反 |
| > 100% | インシデント対応モード / 損保ジャパンへ報告 | SLA クレジット発生確定、再発防止が最優先 |

### 15.4 SLA 違反シナリオ別 推定発生頻度 (リスクモデル)

```
月次 違反確率 (1 ヶ月あたり):
  99.5% 達成 (no クレジット)             : 70-80%  (通常運用)

  98.5-99.0% (10% クレジット)            : 3-5%    (中規模インシデント)
  98.0-98.5% (20% クレジット)            : 1-2%    (Supabase 障害連発等)

## 19. 撤退・縮小・延期の判断基準

### 19.1 判断フロー

```
1. 撤退/縮小トリガを検知
   ↓
2. 24 時間以内に CEO + 経営チーム 会議 (45 分)
   ↓
3. 以下のオプションから選択
   a. 継続 (条件変更なし)
   b. 縮小 (リソース投入を 50% 減、並行 3 社優先)
   c. 延期 (3 ヶ月リセット)
   d. 撤退 (商談クローズ)
   ↓
4. 損保ジャパン側に 1 週間以内に正式通告 (撤退/延期の場合)
   ↓
5. 並行 3 社のうち 1 社を Lighthouse 格上げ判断 (1 ヶ月以内)
```

### 19.2 撤退条件 (どれか 1 つでも該当)

- DPA / 契約で当社が呑めない条項が **3 回交渉して** 解消されない
- 価格 ¥300 万 / 月 を下回る要求が **譲歩されない**
- 「独占的契約」要求が **撤回されない**
- 損保ジャパン側担当が **3 ヶ月で 2 名以上交替**
- 競合が **本番接続を先に達成** + 当社の差別化が成立しない
- ペネトレ 重大指摘の修正が **6 ヶ月以上かかる**

### 19.3 縮小条件 (リソースを 50% に)

- NDA 締結から 6 ヶ月で PoC 範囲書未確定
- フェーズ 2 完了が 3 ヶ月以上遅延
- 加盟店 30 店舗到達が T+180 まで遅延

### 19.4 延期条件 (3 ヶ月リセット)

- 損保ジャパン側で **大型 DX プロジェクト** が並行発生し、リソース不足
- M&A / 経営統合などの **外部要因**
- 当社側で **重大インシデント** (P0 が 30 日内に 2 件以上)

### 19.5 撤退後のアクション

- 並行 3 社のうち 1 社を Lighthouse 格上げ判断
- 「Lighthouse 不要」前提の事業計画への切替判断 (店舗 ARR / OEM ARR で代替)
- 損保ジャパン担当との関係性は維持 (将来再開の可能性)

  - BizDev: 週 30h (75%) → 加盟店拡大が滞る
  - CS: 2 社 = 2 名必要 (年間 ¥1,600 万 増)
  - on-call: 不変だが、社別の SLA 違反確率が上がる

→ 3 社並行を始める前に CS 2 名体制 + BizDev 2 名体制 (¥3,000 万 / 年 増) が必要
