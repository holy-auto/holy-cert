## 1. プラットフォーム概要

Ledraは自動車施工（コーティング・フィルム・ラッピング等）の記録をデジタル証明書として発行・管理するマルチテナントSaaSです。

### 3つのポータル + 顧客ビュー

| ポータル | 対象ユーザー | 主な役割 |
|---|---|---|
| **Admin（施工店）** | 施工店スタッフ | 証明書発行、車両・顧客管理、請求書、予約、BtoB受発注 |
| **Agent（代理店）** | パートナー代理店 | 施工店紹介、コミッション管理、レポート |
| **Insurer（保険会社）** | 保険会社査定担当 | 証明書検索・照会、案件管理、分析 |
| **Customer（顧客）** | エンドユーザー | 自分の証明書閲覧 |

## 2. 施工店ポータル（Admin）

### 2.1 ダッシュボード `/admin`
- パートナーランク表示（プラチナ/ゴールド/シルバー/ブロンズ/スターター）
- KPI カード: 証明書数（合計/有効/無効）、メンバー数、顧客数、請求書数、未回収額
- 予約: 本日の予約数、進行中の予約・作業
- BtoB: 進行中の受発注件数
- 30日間の発行推移チャート、ステータス内訳チャート
- プラットフォーム全体統計（運営権限のみ）: 全体証明書数、業種別・地域別施工店分布
- クイックアクション: 証明書一覧、新規発行、顧客管理、請求書、課金管理
- オンボーディングツアー
- ウィジェットカスタマイズ機能（表示/非表示・並び替え）

### 2.2 証明書管理 `/admin/certificates`
- **一覧**: 検索・フィルタ・ソート、ステータス絞り込み
- **新規発行** `/admin/certificates/new`: 車両情報入力、施工内容記録、写真アップロード、証明書生成
- **発行完了** `/admin/certificates/new/success`: 発行成功画面、QRコード表示
- **詳細** `/admin/certificates/[public_id]`: 証明書詳細閲覧、PDF出力、無効化、複製
- **バッチPDF出力**: 複数証明書の一括PDF生成
- **ステータス変更**: 有効化/無効化
- **証明書複製**: 既存証明書をコピーして新規作成

### 2.3 車両管理 `/admin/vehicles`
- **一覧**: 車両検索・フィルタ
- **新規登録** `/admin/vehicles/new`: 車両情報入力、車検証OCR読取
- **詳細** `/admin/vehicles/[id]`: 車両情報閲覧・編集、証明書一覧、**統合サービス履歴タイムライン**、NFC タグ一覧
  - **サービス履歴タイムライン** (`ServiceTimeline`): `vehicle_histories` + `certificates` + `reservations` + `nfc_tags` を 1 本の時系列に合成して表示。証明書発行・削除、来店・作業開始・完了、NFC 書込を**色分けバッジ**で識別。証明書イベントをクリックすると `/admin/certificates/[public_id]` へ、予約イベントは `/admin/jobs/[id]` へ即遷移
  - 保険査定の「この車両に他の施工はあるか」「いつ何をされたか」の問いに 1 画面で即答できる
  - `vehicle_histories` の新旧スキーマ (`title/description/performed_at` と `label/note/created_at`) を**両対応**して読む
- **CSVインポート**: 車両データ一括取り込み
- **車検証OCR**: カメラ/画像から車両サイズ自動読取

### 2.4 顧客管理 `/admin/customers`
- **一覧**: 顧客検索・フィルタ
- **詳細 (360° ビュー)** `/admin/customers/[id]`: 顧客基本情報 + タブで**車両 / 証明書 / 予約・案件 / 請求**を横断表示
  - 基本情報カード (インライン編集可)
  - タブ内から各詳細ページ・案件ワークフロー (`/admin/jobs/[id]`) へワンクリック遷移
  - 「+ 車両登録」「🪪 証明書発行」「🏃 飛び込み案件」「💰 請求書作成」をタブ右上に常時配置し、
    顧客コンテキストを維持したまま次アクションへ進める
  - データは `Promise.all` で並列取得 (vehicles / certificates / reservations / documents)

### 2.5 予約管理 `/admin/reservations`
- 予約一覧（日付・ステータス絞り込み）
- 予約作成・編集・キャンセル
- ステータス管理: 確認済み → 到着 → 作業中 → 完了
- Google Calendar連携
- 予約の詳細パネルから **案件ワークフロー** (`/admin/jobs/[id]`) へ導線あり

### 2.5b 案件ワークフロー `/admin/jobs/[id]` ★統合ワークスペース
予約 (reservation) を「案件 (Job)」として扱い、**予約 → 来店 → 作業 → 証明書発行 → 請求 → 入金** という一連の業務フローを **1 画面** で進行管理する統合ワークスペース。従来は複数画面に散在していた情報と操作を案件単位で束ねる。

- **ステータスステッパー**: `confirmed → arrived → in_progress → completed` を可視化。1 クリックで次ステータスへ進行 (`PUT /api/admin/reservations`)
- **次アクションパネル**: 現在の状態に応じて「🪪 証明書を発行」「💰 請求書を作成」「👤 顧客詳細」「🚗 車両詳細」へワンタップ。証明書発行は車両ID・顧客IDをURLパラメータとして引き継ぐ (`/admin/certificates/new?vehicle_id=...&customer_id=...`)
- **タブ構成**:
  - **サマリ**: 予約日時 / 概算金額 / 備考 / メニュー一覧
  - **顧客・車両**: 紐付く顧客・車両情報の詳細カード
  - **証明書**: この車両・顧客に紐付く証明書一覧 (public_id, ステータス, 発行日, 施工料金)
  - **請求・見積**: `documents` テーブルから請求書・見積書・領収書・納品書を集約表示
- **重複検知**: 有効な証明書がある場合、請求書が入金済みの場合はボタンに `(発行済)` / `(入金済あり)` を表示し過剰発行を抑止
- **データソース**: `reservations` をハブに `customers` / `vehicles` / `certificates` / `documents` を横断取得 (サーバーコンポーネントで並列 fetch)

### 2.5c 飛び込み案件開始 `/admin/jobs/new`
予約なしで来店した顧客・その場発注の案件を、**1 フォーム / 数秒で** ワークフローに乗せるための軽量 intake ページ。「予約が無い業務はワークフローに乗らない」ギャップを埋める入口。

- **必須入力は案件タイトルのみ** (デフォルトで `飛び込み案件 YYYY-MM-DD` が自動入力)
- 開始ステータスを **🚪 来店・受付 / 🔧 作業中** から選択 (`POST /api/admin/reservations` が `status` を受け取れるよう拡張)
- 顧客・車両は任意。後から `/admin/jobs/[id]` で紐付け可能 (検索 UI で既存顧客を即選択)
- `scheduled_date = 今日` で reservation を作成し、そのまま `/admin/jobs/[id]` にリダイレクト → 通常の案件ワークフローに合流
- **導線**:
  - サイドバー ホーム → ダッシュボードの「🏃 飛び込み案件」クイックアクション
  - 予約管理ページ (`/admin/reservations`) のヘッダーに「🏃 飛び込み案件」ボタン

## 10. 主要ワークフロー

### 10.1 施工証明書発行フロー

```
顧客来店 → 予約チェックイン → 作業開始
    ↓
車両登録（車検証OCR or 手動入力）
    ↓
施工実施 → 写真撮影・アップロード
    ↓
証明書作成（施工内容・写真・備考入力）
    ↓
証明書発行 → QRコード生成
    ↓
顧客にQRコード/URL共有
    ↓
顧客がスマホで証明書閲覧
    ↓
保険会社が証明書を検索・照会可能に
```

### 10.2 BtoB受発注フロー

```
施工店A: BtoBプラットフォームに自店情報掲載
    ↓
施工店B: 得意分野で検索 → 問い合わせ
    ↓
施工店A: 見積もり提示
    ↓
施工店B: 発注
    ↓
施工店A: 受注承認 → 作業実施
    ↓
作業完了 → レビュー
    ↓
支払確認 → 取引完了
    ↓
パートナーランクに実績反映
```

### 10.3 代理店紹介フロー

```
代理店: パートナー申請 → 審査
    ↓
承認 → 紹介リンク発行
    ↓
代理店: 施工店を見つけてLedraを紹介
    ↓
紹介登録（店舗名・担当者・連絡先）
    ↓
施工店: Ledra登録 → 利用開始
    ↓
紹介ステータス更新: 申請中 → 契約成立
    ↓
コミッション発生 → 代理店に支払い
    ↓
ランキング・レポートに反映
```

### 10.4 保険会社 証明書照会フロー

```
保険金請求 → 施工証明書の真正性確認が必要
    ↓
保険会社ポータルにログイン
    ↓
証明書検索（public_id / 顧客名 / 車両情報）
    ↓
証明書詳細確認（施工内容・写真・施工店情報）
    ↓

    ↓
必要に応じてメッセージ送信
    ↓
査定完了 → 案件クローズ
```

### 10.5 予約〜会計フロー

```
顧客: 外部予約 or 直接来店
    ↓
予約登録（日付・時間・施工内容）
    ↓
Google Calendar同期
    ↓
当日: チェックイン → 作業開始
    ↓
作業進捗更新（モバイルAPI）
    ↓
作業完了
    ↓
POS会計 or 請求書発行
    ↓
Square/Stripe で決済
    ↓
売上データ連携 → 経営分析に反映

### 10.6 請求・課金フロー

```
Ledra利用 → 月次課金（Stripe）
    ↓
施工店: 顧客への請求書発行
    ↓

## 12. ロードマップ & UX改善方針

「機能数が多く、導線が分散しがち」という課題に対して、**統合 (consolidation)** と **タスク指向の再設計** を2本柱で進めます。

### 12.1 実施済み (2026-04 時点)

#### サイドバーの 6 カテゴリ再構成
20+ 項目を以下の6グループに集約し、認知負荷を大幅に削減しました (Hick's Law 対応)。

| グループ | 収録項目 |
|---|---|
| (ホーム) | ダッシュボード |
| **業務** | 予約管理 / ワークフロー / 証明書 / 車両管理 / 予約受付設定 / 品目マスタ / NFC管理 |
| **顧客** | 顧客管理 / ヒアリング / 導入ヒアリング / 問い合わせ |
| **売上・経営** | 請求・帳票 / Square 売上 / 経営分析 / POS会計 / 施工価格相場 |
| **取引ハブ** | 取引ダッシュボード / BtoBプラットフォーム / 案件受発注 / マーケット車両 / 商談管理 / 保険会社管理 / 代理店管理 / 代理店ハブ |
| **情報・学習** | お知らせ / 業界ニュース / Ledra Academy / 施工事例 / QAアシスタント / AI添削 |
| **設定** | 店舗設定 / 店舗管理 / メンバー / ブランド証明書 / ロゴ / ショップ / 請求・プラン / 操作履歴 / テンプレ管理 / 運営ダッシュボード |

主な改善点:

- **顧客** を独立グループ化し、「問い合わせ」を顧客コンテキストへ移動
- **Academy** を情報グループに統合 (学習系は情報と親和性が高い)
- **BtoB** → **取引ハブ** へリネーム、商談/マーケット/保険会社管理も統合
- 旧「業務」グループは解体、品目マスタ/NFCを **業務** へ、ヒアリングを **顧客** へ

#### ハブページの新設

| URL | 役割 |
|---|---|
| `/admin/agent-hub` | 代理店向け運用機能 7 画面 (お知らせ/キャンペーン/FAQ/請求書/通知/サポート/研修/営業資料) をカード形式で集約 |
| `/admin/trades` | 取引系機能 (BtoB / 受発注 / 商談 / マーケット車両 / 保険会社) を集約。ファネル (リード→商談→受発注→完了) を意識した導線 |

#### 「案件 (Job)」軸での業務フロー統合 ★実装済み

予約・チェックイン・作業進捗・証明書・請求・決済が別画面で分散していた状態を、**案件ID を串刺しにした1画面** (`/admin/jobs/[id]`) に統合しました。これはハブ型の「入口集約」ではなく、**1 件の案件について業務の流れそのものを具現化する** ワークスペースです。

```
Before: /reservations → /certificates/new → /invoices/new → /pos   (複数画面を往復)
After:  /admin/jobs/[id]  ステッパー: 予約確定 → 来店 → 作業中 → 完了
                         タブ: サマリ / 顧客・車両 / 証明書 / 請求
                         次アクション: 🪪証明書発行 💰請求書作成 (車両ID・顧客IDを引き継ぎ)
```

主な特徴:

- **ステータスステッパー**: `confirmed → arrived → in_progress → completed` を 1 クリックで進行
- **予約 → 証明書 → 請求 の自動リンク**: 車両ID・顧客IDをURLパラメータとして次画面へ引き継ぎ、再入力を排除
- **重複検知**: 有効な証明書がある場合/入金済み請求がある場合にガイド表示で過剰発行を抑止
- **予約詳細パネル**から **🧭 案件ワークフローを開く** ボタンで遷移

#### 飛び込み案件のワークフロー化 ★実装済み

「予約が無い案件 (飛び込み客・その場発注) はワークフローに乗らない」という構造的ギャップを解消。`/admin/jobs/new` で数秒で案件を開始し、そのまま `/admin/jobs/[id]` のワークフローに合流できるようにしました。

- `POST /api/admin/reservations` が `status` パラメータを受け取るよう拡張 (`confirmed` / `arrived` / `in_progress` をホワイトリスト)
- `scheduled_date = 今日`、`status = arrived` で即時 reservation を作成し、`/admin/jobs/[id]` へ自動遷移
- 顧客・車両は後付け OK (空のまま案件開始可能)
- 導線: ダッシュボード クイックアクション + 予約ページヘッダー

#### 遷移ストレスを下げる Loading UX ★実装済み

画面遷移時の「何も起きない間」を極力短く感じさせるため、Next.js 15 の App Router ネイティブ機能を総動員した 3 層のフィードバックを実装しました。Facebook の研究で知られる「スケルトン表示は同じ待ち時間でも体感を 2〜3 倍速く見せる」効果を施工店ダッシュボードに適用しています。

| 層 | 仕組み | 効果 |
|---|---|---|
| ① グローバル進捗バー | `<NavigationProgress>` (画面上部 3px) が `<a>` クリックを検知して即表示、pathname 変化で完了 | クリック→何か起きてる感を 0ms で返す |
| ② ページ `loading.tsx` | 主要画面ごとの骨組みスケルトン (jobs/[id], jobs/new, trades, agent-hub) | ページ全体が空白にならず、レイアウトがブレない |
| ③ `<Suspense>` ストリーミング | `/admin/jobs/[id]` を `JobStatusPanel` (軽量) + `JobTabsLoader` (証明書/書類) に分割 | ステータス操作は先出し、重いクエリはタブ領域だけスケルトン |

実装ポイント:

- **150ms の遅延表示**: 瞬時に終わる遷移でちらつかせない (NProgress 定番の閾値)
- **10 秒 Safety**: フェッチ失敗時も無限にバーが走り続けないように強制終了
- **Promise.all による並列化**: `customer` / `vehicle` / `certificates` / `documents` のフェッチを一斉に飛ばし、順次待ちをやめた
- **スケルトンは `animate-pulse + bg-[rgba(0,0,0,0.06)]`** のトークンに統一し、本物のレイアウトと重なる位置に配置 (視覚的ガイド効果)

#### 顧客 360° ビュー & 車両統合タイムライン ★実装済み

「顧客詳細が編集フォーム＋テーブル縦積み」「車両履歴と予約イベントが別画面」という分散を解消:

- **`/admin/customers/[id]`**: 顧客基本情報カード下に**車両 / 証明書 / 予約・案件 / 請求**の 4 タブを配置。`Promise.all` で並列取得し、タブ右上の「+ 車両登録」「🪪 証明書発行」「🏃 飛び込み案件」「💰 請求書作成」から顧客コンテキストを維持したまま次アクションへ進める
- **`/admin/vehicles/[id]`**: `vehicle_histories` + `certificates` + `reservations` + `nfc_tags` を 1 本の時系列に合成する `ServiceTimeline` コンポーネントを導入。証明書/予約/NFC/削除を**色分けバッジ**で識別し、クリックで各詳細ページへ直接遷移。保険査定の「いつ何をされた車両か」の問いに 1 画面で即答

これにより、ロードマップ 12.2 に記載していた「顧客 360° ビュー」と「車両施工履歴タイムライン」の**土台部分が完了**。LINE 履歴・ヒアリング履歴・メール履歴・音声メモなどの追加ソースはデータスキーマ確立後に差し込み可能な構造。

#### 2 要素認証 (2FA / TOTP) ★実装済み

保険会社・顧客個人情報を扱うアカウントのセキュリティ強化:

- **`/admin/settings/security`**: Supabase Auth の MFA API (`enroll` / `challenge` / `verify` / `unenroll` / `listFactors`) を直接呼び出す専用画面
- TOTP QR コード表示 → 認証アプリで読取 → 6 桁コードで検証 → 有効化
- QR が読めない環境向けにシークレットの手入力フォールバックも提供
- 未検証 (unverified) の factor が残留していた場合はページ初期化時に自動クリーンアップ
- 店舗設定画面 (`/admin/settings`) から「🔐 セキュリティ設定」導線でアクセス

エンタープライズ顧客 (保険会社・大手施工店) との契約時に**単要素認証のみ**が指摘される弱点を解消。

### 12.4 UX改善の基本原則

今後の画面設計で一貫して適用する 4 つの原則:

1. **5±2 ルール** — 1 階層目のメニューは最大 7 項目まで (Hick's Law)
2. **タスク指向 > 機能指向** — 「何をしたいか」から入れる導線設計
3. **データ間リンクを優先** — データ関連性があれば必ずリンク可能に
4. **コマンドパレット中心** — パワーユーザーは Cmd+K で即アクセス可能
